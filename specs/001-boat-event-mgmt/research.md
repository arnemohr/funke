# Research — Funke Event Management System

## Technical Stack

- Decision: Use Python (FastAPI) for backend, deployed on AWS Lambda behind API Gateway.  
  Rationale: FastAPI provides async performance, type hints, and automatic OpenAPI docs; Lambda + API Gateway matches the cost target by scaling to zero.  
  Alternatives considered: Node/Fastify (viable but diverges from Python IaC choice), container-based Fargate (higher baseline cost).

- Decision: Frontend built with Vue 3 (Composition API) on Vite, styled with Pico CSS, with PWA support for offline check-in.  
  Rationale: Vue 3 + Pico CSS deliver a lightweight, maintainable UI with minimal CSS overhead; Vite keeps builds fast for small team velocity.  
  Alternatives considered: React (heavier for simple UI), Svelte (good fit but less standardization in team assumption).

- Decision: Use OpenAPI from FastAPI to generate typed client for Vue (e.g., openapi-typescript) rather than shared code packages.  
  Rationale: Leverages FastAPI’s autogenerated schema to avoid drift while keeping a clean Python/Vue separation.  
  Alternatives considered: Manually maintained DTOs (risk of drift), GRPC/protobuf (overkill for single API).

## Data & Storage

- Decision: DynamoDB as primary store with transactions for capacity checks, lottery writes, and waitlist promotions; TTL for data purge after 90 days.  
  Rationale: Matches serverless cost/scaling goals; transactional API covers capacity and lottery atomicity; TTL handles privacy requirement.  
  Alternatives considered: RDS/Postgres (higher cost/ops), Aurora Serverless v2 (more expensive for low traffic).

- Decision: Use pydantic models for validation and DynamoDB access via boto3 with thin repository helpers; no heavyweight ORM.  
  Rationale: DynamoDB favors explicit access patterns; pydantic keeps request/response validation consistent.  
  Alternatives considered: ORM adapters for DynamoDB (limited maturity), raw client everywhere (more duplication).

## Communications

- Decision: Email via Gmail API with OAuth2, setting Message-ID and In-Reply-To headers; retries via Lambda re-invocation/SQS DLQ with exponential backoff (max 3 attempts).  
  Rationale: Aligns with spec (Gmail API) and avoids SMTP password handling; API supports custom headers.  
  Alternatives considered: SES (cheaper but conflicts with spec), SMTP (more secrets management).

- Decision: Incoming email processing via Gmail API watch + webhook to API Gateway → Lambda to correlate In-Reply-To/Message-ID and append to threads.  
  Rationale: Keeps end-to-end in Gmail API and serverless; enables reply threading requirement.  
  Alternatives considered: IMAP polling (less reliable on Lambda), third-party inbound services (extra cost).

## Lottery & Waitlist

- Decision: Use Python `secrets` with stored seed to shuffle registrations; persist seed and shuffled order for deterministic replay.  
  Rationale: Cryptographically secure randomness and auditable replay.  
  Alternatives considered: `random` (not crypto-strong), external RNG services (complexity).

- Decision: Greedy fill that keeps groups intact; store resulting ordered winners and waitlist with positions.  
  Rationale: Matches spec requirement for group integrity and capacity utilization.  
  Alternatives considered: Fractional allocation (violates requirement), manual override only (no automation).

## Offline Check-In

- Decision: PWA with service worker caching of check-in routes/assets and IndexedDB for offline check-in queue; sync via background sync or manual retry when back online.  
  Rationale: Meets offline requirement with standard web capabilities; no native app needed.  
  Alternatives considered: LocalStorage (insufficient for queue durability), dedicated mobile app (out of scope).

## Security, Rate Limiting, and Auth

- Decision: Admin auth via Auth0; dashboard uses Auth0 SPA flow, backend validates JWTs and enforces roles (Owner/Admin/Viewer).  
  Rationale: Delegates auth to managed provider, reduces custom auth risk for single-tenant admin-only scenario.  
  Alternatives considered: Homegrown password auth (more security surface), Cognito (viable but adds setup friction vs Auth0 per spec).

- Decision: Apply API Gateway rate limiting (10/min/IP) on public endpoints (registration, cancellation, confirmations).  
  Rationale: Built-in throttling meets FR-039 with minimal app code.  
  Alternatives considered: Custom middleware (duplicative), external gateway (unneeded with API GW).

## Deployment & Cost

- Decision: AWS Lambda + API Gateway for backend; S3 + CloudFront for frontend; DynamoDB serverless with on-demand billing; SQS/DLQ for retries if needed; managed Auth0.  
  Rationale: Scales to zero and fits ~€1-5/month given low traffic; aligns with chosen stack.  
  Alternatives considered: ECS/Fargate (higher baseline), self-hosted servers (higher ops, not scale-to-zero).

## Testing & Quality

- Decision: pytest for backend unit/contract tests using FastAPI TestClient; Playwright + Vitest for frontend/E2E covering registration, lottery, offline check-in sync.  
  Rationale: Native tooling per language; covers main risk areas end-to-end.  
  Alternatives considered: Jest (backend mismatch), Cypress (heavier infra).

## Scope & Performance

- Decision: Target 500 concurrent registration attempts with p95 API latency <400ms; email dispatch within 5 minutes; check-in page interactive <2s on 3G with offline cache.  
  Rationale: Matches success criteria SC-005/SC-006/SC-007.  
  Alternatives considered: Higher throughput (unneeded for 1 event/week) or lower targets (would violate success criteria).
